# 仿真文献阅读

## 概念建模框架DEDS

对任何复杂主题进行有意义讨论的关键要求是一系列明确定义的概念，这些概念映射到
感兴趣领域的各个方面。在制定离散事件动态系统模型的背景下，这一要求转化为对框架的需求，该框架提供了一致且连贯的方式来查看和描述引起行为的机制。在本章中，我们提出了这样一个框架。它是我们探索围绕成功完成DEDS领域中任何建模和模拟项目的问题的目标的一个重要方面。

由于DEDS领域的多样性和无限复杂性，尚未出现用于表示该类系统的标准化且普遍接受的框架。各种现有形式，例如有限状态机或Petri网，在特定情况下可能是有用的，但是这些形式缺乏足够的“建模能力”（即通用性）以便普遍适用。另一方面，DEVS方法肯定具有必要的通用性，但由于其基本的数学形式而限制了可访问性。各种选择都是可能的。我们提出的那个是非正式的，具有相当直观的吸引力。但与此同时，它具有高度的通用性和适应性。虽然它可以直接应用于广泛的项目描述，但是当出现特殊需求时，它也可以在临时基础上轻松扩展。

## 概念模型框架的组成部分

我们的DEDS概念建模框架由少量基本组件构成。 第一个是实体的集合，它们在观察间隔过程中通过对我们框架的第二个重要组成部分的事件的发生作出反应并引起事件的发生而相互作用。 实体和事件是基本建筑块。

另外，然而。有一个更高级别的构造称为活动，这是我们的概念建模框架的基础。模型中的一组活动构造提供了捕获相关动态行为的方法。事实上，每项活动都代表了一个需要被识别的特定行为单位。它可以被视为模型中某些实体之间的关系。活动是根据实体和事件的基本构建块来制定的，而这些构件又是根据一些常量，参数和变量的集合来表征的，这些常量，参数和变量的选择是为了以有意义的方式实现这种表征。
我们方法的基本概念并不新鲜。它们可以追溯到活动扫描范例，通常被认为是建模和模拟“世界观”之一。从编程角度全面展示活动扫描可以在Kreutzer找到。需要强调我们基于活动的概念建模框架的一个特殊方面;也就是说，它不能被解释为编程环境。其目的是为程序开发提供有意义的基础，或者说，其目的是作为从项目描述到模拟程序的过渡的工具。



### 实体和模型的结构

C consumer entity instances集合C是不相交的，n是SUI依赖的。 每个消费者实体集C中的cei都是同一个消费者实体类的成员，我们用0表示。

R resource entities : 每个资源实体都与来自C中表示的一些消费者实体类的cei相关。

A aggregate entities :它们具有真实世界的对应物，包括队列和组，后者只是一个没有与队列相关联的排序协议的集合。



cei通常直接与资源和模型中的聚合实体交互。 这种相互作用引起模型状态的变化，这些变化代表了感兴趣的动态行为。 集合C，ft。和Aare取决于SUI的特定性质; 然而，即使是最简单的DEDS的概念模型通常也包括至少一个消费者实体类，一个资源和一个聚合。

资源实体通常可以被视为提供由cei寻求的服务。 因此，模型中的cei通常可以被视为“运动中”，因为它们分散在模型所包含的空间中以寻找可从各种资源实体获得的服务。 由于这些服务可以分配的速度受到限制，寻求服务的人经常不得不等到他们能够得到满足。 等待发生在与资源连接（可能只是隐式）的聚合中。 实际上，我们的概念建模框架通常会产生一种网络结构，其中节点是资源或聚合，链接是cei可以遍历的路径。

在上面给出的描述中隐含的是除了通过连接到聚合或资源之外不存在cei的重要特征。 实际上，模型中的资源和聚合实体共享提供“位置”（可能是虚拟）的重要共同特征，其中cei可以临时驻留。 这种“驻留”通常具有有限的持续时间，其建立和终止都与事件的发生相关联。 如下一节所示，这对我们发展有意义的方式产生了重要影响任何特定模型中各种实体类型的表征。模型的状态变量来自此表征。

还要注意，允许C中的任何特定cei在观察间隔期间仅在模型内具有瞬态存在。 换句话说，消费者实体集C可以是时间依赖的; 也就是说，C_k= C_k(t)。 不排除A和R成员的瞬态存在，但同时也不典型。

举例说明：

为了说明我们概念建模框架的这些基本概念，请考虑百货商店的运作。 顾客到达并且通常打算在商店的各个购物区域进行一次或多次购买。 在每个选定的区域，客户浏览，进行选择，然后在最近的服务台付款，然后再前往下一个购物区。 完成购物任务后，顾客离开商店。

对于项目描述的这个片段，客户对应于消费者实体类，并且在任何特定时刻存储在商店中的那些实例对应于消费者实体集。

各个购物区域中的每个服务台对应于资源。 由于服务台的服务功能持续时间有限，因此有可能在抵达服务台时有些人可能无法立即得到关注。 因此，将队列实体与每个服务台相关联是合理的。 另请注意，每个购物区的浏览阶段都会引入我们称为组的聚合的特殊情况。 从图4.1给出的表示中可以明显看出潜在动态的网络结构。

许多细节尚未澄清; 例如，如何选择特定客户访问的购物区域集？ 访问的顺序是什么？ 服务台分配了多少台服务器？ 特定客户是否可以跟踪，即不在一个或多个指定的购物区域进行任何购买，如果是，那么在什么情况下呢？ 在给定的项目描述片段中没有提供处理这些问题的信息，但在制定有意义的概念模型之前肯定是必要的。 事实上，概念建模过程的一个重要功能是揭示缺乏这些基本细节。
